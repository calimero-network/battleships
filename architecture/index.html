Battleships on Calimero

A multi-service P2P battleship game using namespaces, private storage, and CRDT-based state sync

How It Works

Battleships is a two-player turn-based game built on Calimero. Each game runs across two nodes that sync state automatically via CRDTs. Ship placements are stored privately so your opponent can't see them until they're hit.

The app demonstrates four core Calimero concepts: namespaces for identity and access control, multi-service bundles for separating lobby and game logic, private storage for hidden ship boards, and cross-context calls (xcall) for notifying the lobby when a game ends.

Namespaces & Access Control

A namespace is a root group that scopes identity, governance, and application instances. Each battleships lobby is a namespace.

Hierarchy

namespace (root group)
  // All players join here
  // Per-node Ed25519 identity (auto-generated)
  // Application binding (battleships bundle)

  lobby context — lobby service
    // Matchmaking, player stats, match history
    // All namespace members can access

  match subgroup A — Player 1 + Player 2 only
    match context — game service
      // Private boards, shots, turn state

  match subgroup B — Player 3 + Player 4 only
    match context — game service

Capability Configuration

When the namespace is created, default capabilities are set so every new member inherits:

CAN_CREATE_CONTEXT

Bit 0. Players can create match contexts inside subgroups.

CAN_INVITE_MEMBERS

Bit 1. Players can invite others to the namespace.

MANAGE_MEMBERS

Bit 3. Players can add opponents to match subgroups.

Default capability bitmask: CAN_CREATE_CONTEXT | CAN_INVITE_MEMBERS | MANAGE_MEMBERS = 11

Match Isolation via Subgroups

Each match creates a subgroup nested under the namespace. Only the two players are added. The match context lives inside this subgroup, so only they can execute against it. Namespace admins inherit access for moderation.

Multi-Service Bundle

The battleships .mpk bundle contains two WASM services. Each context specifies which service to run via service_name at creation time.

Lobby Service

struct LobbyState {
  id_nonce: u64,
  created_ms: u64,
  matches: Vec<MatchSummary>,
  player_stats: Vec<PlayerStatsEntry>,
  history: Vec<MatchRecord>,
}

6 methods: create_match, set_match_context_id, get_matches, get_player_stats, get_history, on_match_finished

3 events: MatchCreated, MatchListUpdated, PlayerStatsUpdated

Game Service

struct GameState {
  lobby_context_id: Option<String>,
  active_match: Option<Match>,
}

9 methods: place_ships, propose_shot, acknowledge_shot, get_own_board, get_shots, get_active_match_id, get_current_turn, get_current_user, acknowledge_shot_handler

5 events: ShipsPlaced, ShotProposed, ShotFired, Winner, MatchEnded

Shared Types Crate

Both services share GameError and PublicKey via a battleships-types crate. This crate has no calimero-sdk dependency — only borsh, serde, thiserror, and bs58.

Private Storage

Ship placements are the core secret in battleships. Each player's board is stored using Calimero's private storage system, which is NOT replicated between nodes.

#[calimero_sdk::app::private]
struct PrivateBoards {
  boards: UnorderedMap<String, PlayerBoard>,
}

struct PlayerBoard {
  own: Board, // 10x10 grid with ship positions
  ships: u64, // remaining ship cells
  placed: bool,
}

What’s private vs shared

Private (per-node, not synced)

  • Ship coordinates and board layout
  • Remaining ship cell count
  • Placement status per player

Shared (CRDT-synced)

  • Match state (whose turn, pending shots)
  • Shot boards (hit/miss results)
  • Winner determination
  • placed_p1 / placed_p2 flags

When a shot is proposed, the target player's node resolves it against their private board and publishes only the result (hit/miss). The opponent never sees ship positions until they’re hit.

Cross-Context Calls (xcall)

When a game ends (all ships sunk), the game context notifies the lobby context to update stats and history. This uses Calimero’s xcall — a stringly-typed cross-context function call.

// In game service — acknowledge_shot, after winner determined
calimero_sdk::env::xcall(
  &lobby_context_id, // target context (lobby)
  "on_match_finished", // method name
  &json!({ "match_id", "winner", "loser" }),
);

This works across services because xcall targets a context (by ID), and the lobby context runs the lobby service which has on_match_finished. The method name and JSON params are the only contract between the two services.

Complete Game Flow

1. Create Namespace
Host creates a namespace with the battleships app, sets default capabilities (bits 0+1+3 = 11)
2. Create Lobby
Lobby context created in namespace root group with service_name: lobby
3. Invite Player
Recursive namespace invitation — covers root + all subgroups
4. Player Joins
joinNamespace → auto-gets namespace identity → joins lobby context
5. Create Match
Lobby contract allocates match ID, then: create subgroup → add player2 → create game context with service_name: game
6. Place Ships
Both players place ships — stored in private storage, placed_p1/placed_p2 flags synced
7. Take Turns
propose_shot → event triggers acknowledge_shot_handler on target node → resolves against private board → result synced
8. Game Ends
All ships sunk → winner set → xcall to lobby → stats/history updated

CRDT State Sync

All shared state replicates automatically between nodes using Calimero’s CRDT-based sync. Each mutation creates a delta that’s broadcast via gossipsub and applied on peer nodes.

DAG-based

Deltas form a causal DAG. Concurrent writes merge automatically.

Encrypted

Deltas are encrypted with the group’s shared key before broadcast.

Eventual

Nodes converge to the same root hash. No central coordinator.

The sync layer is service-aware: when applying a delta from a peer, the node loads the correct WASM service (lobby or game) based on the context’s service_name to execute __calimero_sync_next.
